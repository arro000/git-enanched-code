## User Story

**Epic:** EP-006 — CI/CD & Distribuzione
**Priorità:** HIGH | **Story Points:** 3

**Story**
Come maintainer del progetto,
voglio che la pubblicazione su VS Code Marketplace e Open VSX Registry avvenga automaticamente al push di un tag versione,
in modo da rilasciare nuove versioni in modo affidabile e ripetibile senza operazioni manuali soggette a errore.

**Criteri di Accettazione**
- [ ] Un workflow `publish.yml` si attiva al push di tag con pattern `v*.*.*`
- [ ] Il workflow pubblica l'estensione su VS Code Marketplace tramite `vsce publish`
- [ ] Il workflow pubblica l'estensione su Open VSX Registry tramite `ovsx publish`

---

## Soluzione Tecnica

Creare un workflow GitHub Actions `publish.yml` che si attiva esclusivamente su push di tag con pattern `v*.*.*`. Il workflow esegue una pipeline completa di validazione (type-check + test + build) prima della pubblicazione, garantendo che solo codice verificato venga rilasciato. La pubblicazione avviene in sequenza su entrambi i registri utilizzando gli strumenti ufficiali `@vscode/vsce` e `ovsx`.

- **Trigger:** `on: push: tags: ['v*.*.*']` — solo tag di versione semantica attivano il workflow
- **Pipeline di guardia:** type-check → test → build → package .vsix — se un qualsiasi step fallisce, la pubblicazione non avviene
- **Pubblicazione duale:** il file .vsix viene prodotto una sola volta e pubblicato su entrambi i registri (VS Code Marketplace via `vsce publish` e Open VSX via `ovsx publish`)
- **Segreti richiesti:** `VSCE_PAT` (Personal Access Token per Azure DevOps/Marketplace) e `OVSX_PAT` (token Open VSX) configurati come repository secrets
- **Dipendenze dev:** aggiunta di `@vscode/vsce` e `ovsx` come devDependencies in `package.json`

---

## Strategia di Test

Trattandosi di un workflow YAML di infrastruttura CI/CD, non sono necessari test unitari tradizionali. La validazione si concentra su due aspetti: un test strutturale che verifica che il file `publish.yml` esista, sia YAML valido e contenga i job e step attesi; e la verifica che il comando `vsce package` produca un file .vsix valido senza errori.

- Test strutturale del workflow: verifica esistenza file, parsing YAML valido, presenza job `publish`, step `vsce publish` e `ovsx publish`, trigger su tag `v*.*.*`
- Test di packaging: verifica che `npm run package` (vsce package) completi senza errori — questo valida che `package.json` contiene tutti i campi richiesti dal marketplace
- Il workflow stesso include i test del progetto come step di guardia prima della pubblicazione
